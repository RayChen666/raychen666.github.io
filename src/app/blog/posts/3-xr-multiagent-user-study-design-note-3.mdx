## Negotiation Transparency

One of the most compelling aspects of the system identified in our discussions is the **iterative negotiation process** between
the user and the agent during placement. Rather than blindly executing a command, the system surfaces potential conflicts (e.g.,
collision detection, spatial feasibility) and proposes alternatives — capped at **3 iterations per placement phase**.

This design choice is intentional. Transparency in the reasoning process builds **user trust**, which is critical given that
most AI systems are perceived as black boxes. The more users understand *why* a placement is or isn't feasible, the more confident
they feel about the system's decisions. (This part can be added into the SUS question design)

This motivates **User Study Metric 1**: comparing two conditions
 
- **Condition A**: The system reveals partial negotiation reasoning (collision warnings, layout constraints, etc.)
- **Condition B**: The full reasoning process is blacked out from the user's view
 
The dependent variable is **user confidence and perceived usability**, measured via a standardized usability scale (e.g., SUS)
combined with post-task interviews.

## Graph-Based vs. Chain-Based System

The second core innovation being evaluated is the **graph-based multi-agent architecture** versus a purely sequential,
chain-based prompt pipeline. The graph-based approach enables:
 
- **Parallel execution** of agents (e.g., asset agent and scene agent operating simultaneously)

- **Memory agent support** for multi-modal input (images, JSON, text)

- **Complex reasoning** for vague or multi-step commands

- **Higher placement precision**, especially for spatially ambiguous instructions
 
This motivates **User Study Metric 2**: a comparative efficiency and effectiveness study between the graph-based system
and a chain-based baseline, measuring **task completion time**, **error rate**, and **number of iterations** required per command.
 
Based on the discussion result btw me and Keru, we decide to design a Quantitative user study to compare the effect. We provide example 
scene to the users, and set the world to be plain or with some random assets inside. We count how many steps does it take for users to 
replicate the example scene we provided btw 2 systems. (we meaure the #step & commands taken, system response rate, corrctness, etc., and 
make those measurements into charts and tables)

The third axis of the user study examines **different modalities of user–system interaction**. The primary interaction modes
under consideration are:
 
- **Natural language** (voice or text commands)

- **Gaze-based** interaction (not the focus, will not develop this interaction feature)

- **Hand gesture** input (probably, but not the main focus, may just use for user study comparison)
 
For the purposes of this study, natural language remains the **main interaction mode**, since the core contribution is the
agent pipeline's ability to parse spatial intent from language. We need to design the tasks that force users to use the natural 
language to complete in order to prove the innovation of the system design. 

## Interview & Survey Design
 
Following Keru's guidance, a complete HCI study should include both quantitative and qualitative components. The post-task
interview questions are designed around:
 
- Which features of the system caused friction or required extra iterations?

- At which stage did the user feel "stuck," and why?

- Can users articulate their satisfaction with the placement outcome?
 
These open-ended questions are meant to surface the aspects of the workflow that prevent the system from becoming a seamless,
trustworthy tool — not just one that technically works, but one that users *want* to use.
