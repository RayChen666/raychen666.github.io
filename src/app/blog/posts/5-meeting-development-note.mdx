## Voice Input

There are three available options for the voice input as Saurarbh and I proposed in our last meeting: 
Headset Direct Input (Meta Quest built-in), Web Speech API (Browser Voice Recognition), and Gemini API, each has its 
own Pros and Cons. 

**A. Headset Direct Input**

**Pros:**

- Lowest latency — processed on-device or via Meta's own pipeline

- No extra API calls in the chain

**Cons:**

- Tightly coupled to Meta's ecosystem — breaks portability to other headsets

- Limited programmatic control

Conclusion: Not viable for our browser-based WebXR system. The OS-level voice input isn't cleanly 
accessible from a web context.

**B. Web Speech API**

This is the `webkitSpeechRecognition` / `SpeechRecognition` API built into browsers.

**Pros:**

- Zero cost — completely free

- Native to the browser, no extra dependencies

- Works in WebXR context since it's just a browser API

- Simple to implement

**Cons:**

- Chrome only in practice — Firefox and Safari support is inconsistent 

- Requires internet connection

- No custom vocabulary — technical terms like object names may misfire

**C. Gemini API**

Using a cloud speech-to-text API explicitly — either Gemini's multimodal 
input, or Google Cloud Speech-to-Text.

**Pros:**

- Best accuracy, especially for technical/accent-specific vocabulary

- Gemini specifically can handle speech + context together

- Configurable — custom vocabulary, confidence thresholds, language 
model priming

**Cons:**

- Added latency — network round trip for every utterance

- Cost — API calls add up, especially during user studies

- Requires audio capture pipeline in WebXR, which adds implementation complexity

**Proposed Solution:**

At first we finished the first worable demo with web speech API, as it is fast to 
implement, zero friction, good enough for controlled lab conditions, where we 
control the environment and can speak clearly. Also, using it to get the rest of the 
pipeline to work first without being blocked by audio infrastructure makes sense, as building 
audio infra takes a lot of time and is not our main research goal.

**Gemini for the complex route specifically**

Since our complex route already involves a heavier reasoning step in the memory agent, 
the extra latency of a Gemini API call seems acceptable here. Also, Gemini's ability to take 
audio + scene context together could actually collapse the voice recognition + language agent 
steps into one call for complex commands, which is an interesting optimization, but this requires 
further development (depends on the time availability).
