## Issues Identified

**Counter-Intuitive Action**

<figure style={{ maxWidth: '768px', margin: '1.5rem auto' }}>
   <img src="/images/blog/12-meeting-note-4/counter-intuitive-action.jpg" 
    alt="Control panel layout" 
    width="100%"
    style={{ borderRadius: '20px' }}
  />
</figure>

When user say "place a chair onto the table", the system undersatand the meaning and do the job correctly. 
When user say "let the table onto the chair", the system also process the command by placing the table on top of chair. 
However, this is counter-intuitive in the real-world situation. This reflects a problem that the system now does not have 
the physical knowldge of the **real-world physics**. 

We are not aiming to solve it at the current development stage, as we are fulfilling the basic functionalities of 
the world creation. Since real-world users will not think about adding a table and place it on top of the chair in 
general cases. However, if we are arguing our system as a spatially coherent system, then it is critical to mention this as a 
bullet point that the system understand real-world physics to some extent. 
As a result, it worth opening an Issue in the GitHub page.

**Lack of Bounding Box Collision Check**

<figure style={{ maxWidth: '768px', margin: '1.5rem auto' }}>
   <img src="/images/blog/12-meeting-note-4/collision-check.jpg" 
    alt="Control panel layout" 
    width="100%"
    style={{ borderRadius: '20px' }}
  />
</figure>

This image shows two commands used in conjunction, and multiple spatial failures.

- We did, "Add a table", "Add a Chair on top of the table”, “Remove the table”. What happens once the table is removed 
successfully? We now see a floating chair! This should be an invalid action, yet because the system doesn’t really 
understand the relationship of objects, especially in the Z-Offset, it removes things indiscriminately.

- Next, “Add four chairs in a straight line”, Causes one of the chairs to clip with the left wall. This might be happening 
because The VerificationNode doesn’t check for collision with the walls themselves.

**Incorrect Facing Direction**

<figure style={{ maxWidth: '768px', margin: '1.5rem auto' }}>
   <img src="/images/blog/12-meeting-note-4/facing-direction.jpg" 
    alt="Control panel layout" 
    width="100%"
    style={{ borderRadius: '20px' }}
  />
</figure>

Following case of “Add a table”, “Add four chairs surrounding the take with inward facing orientation” is a fascinating 
use case of how we could encode the “orientation factor” into the SceneAgent. We know what the “front” part of the chair 
is. But does the LLM know that? It would - if we give it the correct context, at the cost of added complexity.
