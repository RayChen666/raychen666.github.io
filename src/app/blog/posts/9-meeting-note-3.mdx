## Complex Route Design update

Extending from the listerature review discussion on LayoutGPT article, we agreed that LayoutGPT is an excellent proof of concept for your JSON 
based spatial input (for which we already see great results)

However, the goal is to improve the Vague/ Complex route to interpret more accurately. We discussed literature reviews, which referenced novel 
spatial approaches using LLMs such as LayoutGPT, which says that sometimes a structural written prompt can help LLMs understand layouts much 
better than image recognition, owing to the fact that purely Transformer based (ViT) models might not be the best at object detection. This 
in turn becomes a Computer Vision problem, where we would have to explore objection detection, segmentation, semantic object detection etc., 
which is not in the best interest of our LLM focused research.

## Image Agent Design

We proposed that we could use reference images of physical spaces such as Ofﬁces, Cafes, and Homes, as input to a separate agent, to help 
nudge the overall MAS-XR in the right direction. This means, we would have a strong Semantic Analyzer, as opposed to a purely spatial analysis.

**Question:** What is the Semantic approach in this system?

**Answer:** Currently the system has ﬁve Agents, of which four are purely spatial analysts, and while the Language Agent does understand “intent” 
and “semantics”, it’s purely meant to analyze user command in natural language. That means to solve complex problems such as “turn my 
bedroom into a minimalist ofﬁce room”, we might need a Semantic Analyst Agent, that tries to understand the intent of the user based 
on a “aesthetic goal”, (more visually!).

**Question:** How is the memory module / agent process the image and generate the sumantc layout plan for scene and asset agent to 
understand?

**Answer:** separate the image processing function to another image agent

The current design for thr memory module is like:

This means, we separate step 2, 3, and 4.A to form an image agent, while the memory module import the processing 
result of the image agent and only do step 1, 4.B, and 5.

## AI Hallucination

One issue Saurabh noticed and discussed we me is the **"LLM Hallucination"**. To understand, check links below:

- **Medium post:** [LLM Hallucinations Explained](https://medium.com/@nirdiamant21/llm-hallucinations-explained-8c76cdd82532)

- **IBM blog:** [What Are AI Hallucinations?](https://www.ibm.com/think/topics/ai-hallucinations)

- **ACM TOIS conference paper:** [A Survey on Hallucination in Large Language Models](https://arxiv.org/abs/2311.05232)

To connect this to our use case and system design, we raised a potential symptom case: when we look at the top-to-bottom view of the 
office layout (see picture below), we, as human, clearly know that the 3 triangle-shapes are windows due to our life experience and 
logical reasoining. However, for AI with hallucination, it is likely that they mark those as doors or some furnitures. (It is impossible 
for an office to have 3 dorrs in one-side wall, and this is intuitive to human).

## Graph-Based Data Structure

**Graph Adjacency Lists**

In classic Algorithms, a graph is represented as either an adjacency list or an Adjacency Matrix. Saurabh proposed, an adjacency list is as simple 
as making all the objects in the scene as “nodes” in the graph, and the adjacency list will have the objects relationship with all other objects 
(or the one’s closer to it). Refer to following example derived from classic DSA:

<figure style={{ maxWidth: '768px', margin: '1.5rem auto' }}>
   <img src="/images/blog/9-meeting-note-3/adjacency-list.jpg" 
    alt="Control panel layout" 
    width="100%"
    style={{ borderRadius: '20px' }}
  />
</figure>
In above example, if we represent “0”, “1”, “2”, and “3” with the names for objects in the scene, the adjacency list may be written out in JSON as:

<CodeBlock
  marginBottom="16"
  style={{ borderRadius: '20px' }}
  codes={[
    {
      code: `
[
    {
      "object-name": "sofa",
      "adjacency": [
        { "neighbor": "chair", "direction": "right", "distance": "1 meter" },
        { "neighbor": "table", "direction": "left", "distance": "2 meters" }
      ]
    },
    {
      "object-name": "chair",
      "adjacency": [
        { "neighbor": "sofa", "direction": "left", "distance": "1 meter" }
      ]
    }
]
`,
      language: "JSON",
      label: "JSON"
    }
  ]}
/>

**Note:** It is crucial to understand that while the representations are not novel, using an LLM to make decisions based on this structure might be 
novel. Further literature review studies would shed light on this proposal’s usefulness. It is also important to note that it will also signiﬁcantly 
increase the LLM Agent Token usage with the Image and Spatial Graph based context input if this Semantic Route is ever implemented.

**SceneGraph**

Due to Saurabh’s entry level experience with game engines, he proposed another way to represent the object hierarchy. A scene 
graph is a hierarchical tree data structure used in 3D engines and graphics software (like Unity or Open 3D Engine) to organize 
graphical scenes. Attached are some images that may help us understand SceneGraphs as a possible representation for our objects.

Read the study here for more info made by RPI: ➔ [Scene Graph Generation](https://sites.ecse.rpi.edu/~cvrl/SG/SG.html)

The study showed an example which might help us understand how SceneGraphs work (firgure below shows an image with different 
objects and its scene Graph):
